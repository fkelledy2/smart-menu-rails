# Strikepay Integration — Direct Staff Tips

## Status
- Priority Rank: #31
- Category: Post-Launch
- Effort: L
- Dependencies: `Payments::Orchestrator` (existing), Strikepay API access and developer agreement, `Employee` model (existing), `Ordr` model (existing)

## Problem Statement
Restaurant staff in markets where direct digital tipping is customary (Ireland, UK, US) have no mechanism to receive tips via mellow.menu. Tips currently bypass the platform entirely. Integrating Strikepay — a direct-to-staff tipping platform — allows customers to tip individual staff members at checkout or via a staff QR code, with tips flowing directly to staff personal accounts rather than through restaurant revenue. This improves staff retention, differentiates mellow.menu for premium venues, and creates a new data signal (tip rates per staff member) that can feed future analytics features.

## Success Criteria
- Staff members can link their Strikepay account to their mellow.menu employee profile
- Customers can tip a staff member from the SmartMenu post-order screen via a QR code or Strikepay link
- Tip transactions are recorded in mellow.menu for reporting purposes, even though payment flows through Strikepay directly
- Restaurant managers can view a tip summary (total tips per staff member, per period) in the restaurant dashboard
- The integration does not route tip money through mellow.menu's Stripe/Square accounts — tips go directly to the staff member via Strikepay

## User Stories
- As a server, I want to link my Strikepay account to my mellow.menu profile so customers can tip me digitally via a QR code.
- As a customer, I want to easily tip my server from the SmartMenu post-payment screen without needing a separate app.
- As a restaurant manager, I want to see how much each staff member has received in tips this week for reporting purposes.
- As a mellow.menu platform admin, I want to monitor Strikepay API errors and integration health across restaurants.

## Functional Requirements
1. Staff members can connect a Strikepay account from their employee profile settings (`/restaurants/:id/profile` or equivalent), gated by the `strikepay_tipping` Flipper flag.
2. The connection flow uses Strikepay's OAuth or API key flow to associate a staff member's Strikepay account ID with their `Employee` record.
3. A `StrikepayAccount` record is created in mellow.menu linking the `Employee` to their Strikepay account. No sensitive credentials are stored — only the Strikepay account ID and a display name.
4. Post-payment SmartMenu screen shows a "Tip your server" section if the serving employee has a linked Strikepay account. This section displays the employee's name and a link or QR code that opens their Strikepay tipping page.
5. When a customer follows the Strikepay link and completes a tip, Strikepay sends a webhook to mellow.menu (`POST /webhooks/strikepay`). The webhook creates a `StrikepayTip` record for reporting.
6. The mellow.menu platform does not process or hold tip funds at any point — all money flows through Strikepay directly. mellow.menu only stores webhook-confirmed tip records for reporting.
7. Restaurant managers see a tip summary in the restaurant analytics: total tips per employee, per day/week/month.
8. Restaurant owners can disable Strikepay tipping for their restaurant entirely from restaurant settings.
9. The "Tip your server" section is only shown when the specific serving employee (if recorded on the `Ordr`) has a linked Strikepay account; otherwise the section is hidden.
10. Tip QR codes are generated by Strikepay and served as a URL — mellow.menu does not generate or store QR image files.

## Non-Functional Requirements
- **Critical architecture rule**: All payment-adjacent calls go through `Payments::Orchestrator`. If direct Strikepay calls are needed, add a `Payments::StrikepayAdapter` that is invoked exclusively via the Orchestrator. Never call the Strikepay API directly from a controller or model.
- Strikepay webhook endpoint must verify the webhook signature (HMAC or similar, per Strikepay's docs) before processing any payload. Use a `StrikepayWebhookVerifier` service.
- Frontend: Turbo Frames and Stimulus only — no React. The post-payment tipping section is a Turbo Frame loaded asynchronously after the payment success screen renders.
- No PII about the customer is passed to Strikepay via mellow.menu — the customer interacts with Strikepay's own payment UI directly. mellow.menu only records tip confirmation webhook data.
- Tip pool functionality (splitting tips between multiple staff members) is out of scope for v1 — this involves financial distribution logic that requires separate compliance review.

## Technical Notes

### Architecture — why NOT through Payments::Orchestrator's existing Stripe/Square path
Strikepay operates an entirely separate payment rail from the restaurant bill. Tips go staff-member → Strikepay → employee's bank/wallet, not through the restaurant's Stripe or Square account. Therefore Strikepay does not require Stripe integration within mellow.menu. However, API calls to Strikepay must still be made through a dedicated adapter (`Payments::StrikepayAdapter`) to maintain the single-responsibility principle and keep all third-party payment API calls in one place.

### New model: StrikepayAccount
```ruby
create_table :strikepay_accounts do |t|
  t.references :employee,              null: false, foreign_key: true
  t.references :restaurant,            null: false, foreign_key: true
  t.string     :strikepay_account_id,  null: false
  t.string     :display_name,          null: false
  t.boolean    :active,                null: false, default: true
  t.timestamps
  t.index [:employee_id, :restaurant_id], unique: true
  t.index :strikepay_account_id
end
```

Do NOT store `strikepay_api_key` or `strikepay_api_secret` on the restaurant or employee record. API credentials are environment-level (Strikepay uses a platform API key, not per-restaurant keys, if they support marketplace/platform mode). Confirm with Strikepay before designing auth.

### New model: StrikepayTip
```ruby
create_table :strikepay_tips do |t|
  t.references :strikepay_account,     null: false, foreign_key: true
  t.references :ordr,                               foreign_key: true  # optional
  t.string     :strikepay_tip_id,      null: false, index: { unique: true }
  t.decimal    :amount,                null: false, precision: 10, scale: 2
  t.string     :currency,              null: false, default: 'EUR'
  t.string     :status,                null: false, default: 'confirmed'
  t.datetime   :tipped_at,             null: false
  t.timestamps
  t.index :strikepay_account_id
  t.index :tipped_at
end
```

### Service: Payments::StrikepayAdapter
`app/services/payments/strikepay_adapter.rb`:
- `account_info(strikepay_account_id)` — fetches account details for display
- `tipping_url(strikepay_account_id)` — returns the customer-facing Strikepay tipping link for an employee
- Called via `Payments::Orchestrator` — add `:strikepay` as a provider option

### Webhook handler
`app/services/payments/webhooks/strikepay_ingestor.rb` (mirrors existing `StripeIngestor` pattern):
- Verifies HMAC signature
- Handles `tip.confirmed` event: creates `StrikepayTip` record
- Handles `tip.refunded` event: updates `StrikepayTip#status` to `refunded`
- All processing is done synchronously in the webhook controller for immediate acknowledgement; heavy work is delegated to `ProcessStrikepayWebhookJob`

### Background job
`app/jobs/process_strikepay_webhook_job.rb` — processes the webhook payload, creates `StrikepayTip`, triggers any notifications.

### Pundit policy
`app/policies/strikepay_account_policy.rb`:
- `connect?` — employee is managing their own record
- `view_tips?` — employee viewing own tips, or manager/admin viewing any employee at their restaurant
- `disable?` — restaurant admin only

### Flipper flag
- `strikepay_tipping` — master switch for the entire feature; gates both the employee settings connection flow and the SmartMenu post-payment tip prompt

## Acceptance Criteria
1. A staff member can link their Strikepay account from their employee profile; a `StrikepayAccount` record is created and `active: true`.
2. After a successful order payment on SmartMenu, if the serving employee has an active Strikepay account, a "Tip your server" section appears with a Strikepay link.
3. If the serving employee has no Strikepay account, the tipping section is hidden and the post-payment screen is unaffected.
4. A Strikepay webhook with event `tip.confirmed` creates a `StrikepayTip` record. A duplicate webhook (same `strikepay_tip_id`) is idempotently ignored.
5. A Strikepay webhook with an invalid HMAC signature is rejected with a 401 response and no database record is created.
6. The restaurant analytics dashboard shows correct tip totals per employee, calculated from `StrikepayTip` records.
7. A restaurant admin disabling Strikepay tipping in restaurant settings hides the tipping section for all employees at that restaurant, even if individual `StrikepayAccount` records remain active.
8. The `strikepay_tipping` Flipper flag, when disabled, hides all tipping UI and returns 404 for the webhook endpoint.
9. No Strikepay API call is made directly from a controller — all calls go through `Payments::Orchestrator` → `Payments::StrikepayAdapter`.
10. No customer PII is stored in the `StrikepayTip` record — only amount, currency, timestamp, and tip ID.

## Out of Scope
- Tip pooling / distribution across multiple staff members (v2 — requires compliance review)
- Staff-initiated tip requests (push notifications to customers)
- Payout scheduling or staff payroll integration
- Integration of Strikepay tips into the restaurant's Stripe/Square revenue reporting
- In-app wallet or direct bank transfer for tips

## Open Questions
1. Does Strikepay offer a platform/marketplace API mode where mellow.menu acts as the platform and restaurants/staff are sub-accounts? Or does each staff member need a standalone Strikepay account? This fundamentally affects the auth design. **Must be confirmed with Strikepay BD before any development begins.**
2. What markets is Strikepay available in? The integration should only be offered to restaurants in supported markets.
3. Is tip income reporting (for tax purposes) Strikepay's responsibility or mellow.menu's? Legal/compliance confirmation needed before launch.
4. Does Strikepay provide webhook delivery guarantees (retries, at-least-once)? The `StrikepayTip` idempotency key approach handles duplicates but requires Strikepay to retry on failure.

## Pre-Development Gate
Do not begin development until Question 1 (platform API mode vs standalone accounts) is resolved. The entire auth architecture depends on this answer.
